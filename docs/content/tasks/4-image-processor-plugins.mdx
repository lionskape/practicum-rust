# Проект модуля 4. Обработчик изображений с плагинами

> **Примечание:** в оригинальном задании сказано, что приложение принимает четыре аргумента командной строки. Ниже в самом ТЗ перечислены четыре обязательных аргумента (`input`, `output`, `plugin`, `params`) и дополнительный аргумент `plugin-path` со значением по умолчанию `target/debug`. В этой странице сохранена именно фактическая схема из задания.

## Инфраструктура проекта

> Проект разрабатывается в существующем репозитории с уже настроенной инфраструктурой:
> - **Cargo workspace** — корневой `Cargo.toml` с `members = ["crates/*", "xtask"]`
> - **xtask** — система автоматизации сборки с алиасами в `.cargo/config.toml`
> - **CI** — `cargo ci` запускает форматирование, clippy и тесты
> - **Документация** — Nextra-сайт в `docs/`
> - **Nightly toolchain** — проект использует общий toolchain репозитория
>
> Новые крейты создаются в директории `crates/` и автоматически подхватываются workspace через glob `crates/*`.

## Описание

Цель проекта — разработать CLI-приложение для обработки изображений с поддержкой динамически подключаемых плагинов через FFI.

Выполняя этот проект, вы закрепите на практике:

- работу с изображениями в Rust;
- динамическую загрузку и использование библиотек;
- взаимодействие через FFI и осознанное применение `unsafe`-кода;
- проектирование расширяемой плагинной архитектуры;
- обработку ошибок и безопасное управление памятью.

## Что нужно сделать

Нужно реализовать приложение, которое:

- загружает PNG-изображение;
- динамически подключает плагин обработки;
- передаёт изображение в плагин через FFI;
- сохраняет обработанный результат в PNG.

В рамках этого репозитория удобно оформить решение как набор крейтов в `crates/`:

- `image_processor` — основное CLI-приложение;
- `mirror_plugin` — динамическая библиотека для зеркального отражения;
- `blur_plugin` — динамическая библиотека для размытия.

## Задание

Вам предстоит написать CLI-приложение, которое загружает изображение, применяет к нему указанный плагин обработки и сохраняет результат.

### Архитектура приложения

Основной бинарный крейт должен:

1. Принимать аргументы командной строки.
2. Загружать и декодировать изображение.
3. Динамически загружать плагин.
4. Передавать данные плагину.
5. Сохранять результат обработки.

### Аргументы командной строки

Рекомендуется использовать крейт `clap`.

- `input` — путь к исходному PNG-изображению.
- `output` — путь, по которому будет сохранено обработанное изображение.
- `plugin` — имя плагина без расширения.
- `params` — путь к текстовому файлу с параметрами обработки.
- `plugin-path` — путь к директории, где находится плагин. Значение по умолчанию: `target/debug`.

Пример запуска:

```bash
cargo run -p image_processor -- \
  --input ./input.png \
  --output ./output.png \
  --plugin mirror_plugin \
  --params ./params.json \
  --plugin-path ./target/debug
```

### Загрузка и обработка изображения

Для работы с изображением нужно использовать крейт `image`.

Приложение должно:

- читать PNG-файл;
- преобразовывать его в формат `Rgba8`;
- получать ширину, высоту и плоский массив пикселей RGBA;
- после обработки собирать изображение обратно и сохранять его в PNG.

### Динамическая загрузка плагинов

Для загрузки динамических библиотек нужно использовать крейт `libloading`.

Приложение должно:

- находить файл библиотеки по имени плагина;
- учитывать платформенный формат имени библиотеки:
  - `lib<name>.so` на Linux,
  - `<name>.dll` на Windows,
  - `lib<name>.dylib` на macOS;
- загружать из библиотеки функцию `process_image` с ожидаемой сигнатурой.

> Если имя пакета содержит дефисы, итоговое имя библиотеки всё равно будет с подчёркиваниями. Поэтому для плагинов в этом проекте проще использовать имена вроде `mirror_plugin` и `blur_plugin`.

### Передача данных плагину

Плагину нужно передать:

- ширину изображения;
- высоту изображения;
- указатель на буфер RGBA;
- строку параметров, прочитанную из файла `params`.

Формат файла с параметрами может быть любым текстовым: JSON, `key=value`, TOML-подобный формат и т. п.

### API плагина

Все плагины должны экспортировать одну функцию со следующей сигнатурой на языке C:

```c
void process_image(
    uint32_t width,
    uint32_t height,
    uint8_t* rgba_data,
    const char* params
);
```

Описание параметров:

- `width`, `height` — размеры изображения в пикселях.
- `rgba_data` — указатель на массив байтов размером `width * height * 4`.
- `params` — строка параметров. Плагин должен самостоятельно её распарсить.

Важные детали:

- плагин модифицирует данные на месте, в том же буфере `rgba_data`;
- плагин не должен выходить за границы массива;
- для простоты можно считать, что плагины написаны на Rust и собираются как `cdylib`;
- модуль `plugin_loader` должен использовать `libloading` для загрузки библиотеки и получения указателя на `process_image`.

### Обработка ошибок

Нужно обрабатывать следующие ситуации:

- входной файл не существует;
- файл параметров не существует;
- библиотека плагина не найдена;
- функция `process_image` не найдена в библиотеке;
- изображение не удалось загрузить из-за неверного формата или повреждённых данных.

Важно гарантировать, что переданные плагину указатели валидны и память не освобождается до завершения вызова плагина.

### Плагины

Для демонстрации работы системы нужно реализовать как минимум два плагина.

#### Плагин зеркального отражения

Плагин должен принимать два булевых параметра:

- `horizontal` — отражение по горизонтали;
- `vertical` — отражение по вертикали.

#### Плагин размытия

Плагин должен принимать параметры:

- `radius` — радиус размытия;
- `iterations` — количество итераций.

Алгоритм размытия можно выбрать самостоятельно. Подойдёт и простая реализация, в которой новый цвет пикселя рассчитывается по соседним пикселям в пределах радиуса размытия.

## Требования к результату

### Сборка и структура проекта

- В репозитории есть бинарное приложение и хотя бы один плагин.
- Проект компилируется без ошибок.
- Плагины собираются как `cdylib`.
- В зависимостях проекта присутствуют `image`, `clap`, `libloading`.
- Код разделён на логические модули: как минимум `main.rs`, `error.rs`, `plugin_loader.rs`.
- `README.md` содержит описание запуска и примеры команд.

### Аргументы командной строки

- Приложение принимает аргументы `input`, `output`, `plugin`, `params`, `plugin-path`.
- Для парсинга аргументов используется `clap` или аналог.
- При отсутствии аргументов выводится справка.
- Аргументы валидируются до начала обработки.

### Загрузка и обработка изображения

- Изображение загружается через `image`.
- Оно корректно преобразуется в `Rgba8`.
- Ширина и высота извлекаются без ошибок.
- Буфер пикселей доступен как плоский RGBA-массив.
- После обработки изображение сохраняется обратно в PNG.

### Динамическая загрузка плагинов

- Используется `libloading` или аналогичный крейт.
- Имя библиотеки определяется с учётом текущей ОС.
- Символ `process_image` загружается с корректной сигнатурой.
- Ошибки загрузки библиотеки и символа обрабатываются.

### Безопасность FFI

- Вызов `process_image` заключён в `unsafe`-блок.
- Указатель на данные получен безопасно, например через `as_mut_ptr()`.
- Размер буфера вычислен корректно: `width * height * 4`.
- Строка параметров преобразуется в `CString`.
- Память не освобождается до завершения работы плагина.
- После вызова плагина ресурсы освобождаются корректно.

### Примеры плагинов

- Плагин зеркального отражения реализован и работает.
- Плагин размытия реализован и работает.
- Плагины корректно читают параметры из `params`.
- Плагины не выходят за границы переданного буфера.
- Для основной логики плагинов есть тесты.

### Обработка ошибок

- Все операции с файлами и библиотеками возвращают `Result`.
- Ошибки выводятся пользователю в понятном формате.
- Приложение не паникует на некорректных входных данных.
- Ресурсы освобождаются корректно.

### Дополнительно

Рекомендуется:

- парсить параметры плагина в структуру, например из JSON;
- использовать `log` и `env_logger` вместо `println!` для отладочных сообщений.
